# Stirrup v0.1 Release Plan

Status: draft for the v0.1.0 cut. Audit date 2026-06-06. Tested commit
`0191fecb9e1b85e770c357b6e1704b15083fe01a` (branch `0-1-prep`, working tree
clean). No git tags exist yet; v0.1.0 will be the first release.

This plan is written in the project's impersonal, instructional voice. Evidence
is cited as `file:line` or `workflow + run date` throughout. Where the input
data is thin, the claim is marked **unverified** rather than assumed.

---

## 1. Executive summary

**Readiness verdict: conditionally ready.** The shipped functionality is
strongly covered by automated tests — a full local suite (4136 tests, 0
failures) and a continuously-green CI verify gate substantiate the large
majority of v0.1 claims. The blockers are not in the unit-tested core; they
cluster in two places: (a) a small set of **doc/code defects** surfaced by the
adversarial audit that must be fixed or reworded before the claims ship, and
(b) a band of **never-executed live evidence** — four GHA smoke workflows have
zero runs ever, and the entire Cloud Run / GAR / release pipeline has never
been exercised.

**Headline counts (post-audit, 161 falsifiable claims across 9 areas):**

| Coverage | Count |
|---|---|
| verified | 107 |
| partial | 50 |
| unverified | 4 |

Two original classifications were overturned by the skeptic and are reflected
above: `core-17` (secret scrubber "structurally impossible" overclaim) was
downgraded verified -> partial; `executors-10` stays partial but its **claim
text is factually wrong** (k8s exec cap is 10 MB, not "matching the container
executor", whose exec cap is 1 MB).

**Top risks (ranked):**

1. **Live provider conformance is proven for Anthropic only.** Bedrock, Azure
   OpenAI, and Vertex Gemini smoke workflows have never been dispatched
   (`smoke-bedrock.yml`, `smoke-azure-openai.yml`, `smoke-vertex-gemini.yml`
   all zero runs); the GCP WIF STS/impersonation endpoints have never been hit
   live (`federation-18`).
2. **The release + Cloud Run pipeline is entirely unexercised.** `release.yml`
   has zero executions; dual-registry publish to GAR, SBOM upload, cross-compile
   matrix, prerelease marking, and `smoke-cloud-run-job.yml` have never run.
3. **Two real defects behind shipped claims.** Eval `file-contains`/`file-exists`
   judges are **not** symlink-safe despite the doc (`eval-framework-04`, a
   workspace-escape read); `STIRRUP_FOLLOWUP_GRACE` on the `stirrup job` path is
   **not** capped despite the doc's "Capped at 3600 s" (`deployment-18`).
4. **K8s hardening is documented and source-correct but untested by automation.**
   `AutomountServiceAccountToken=false`, the five `securityContext` fields, and
   NetworkPolicy-before-Pod ordering have no cluster-free assertion
   (`executors-01/02/03`); NetworkPolicy enforcement depends on the cluster CNI
   (`executors-16`).
5. **A documented security footgun has zero coverage.** `OTEL_EXPORTER_OTLP_HEADERS`
   bypasses the headers-require-HTTP validator and can ride plaintext gRPC
   (`observability-17`).

None of the four `unverified` claims block a 0.x alpha if the alpha contract
(Section 6) disclaims them honestly; the defects in risk 3 and the doc overclaims
do block, because shipping them as written would be inaccurate.

## 6. Tagging & release mechanics

### 6.1 Multi-module workspace tagging

Stirrup is a `go.work` (go 1.26.1) workspace of four modules. Verified
dependency graph:

- `types` — leaf, no internal deps.
- `gen` — leaf, no internal deps (proto-generated).
- `harness` — requires `gen` **and** `types` (currently `v0.0.0` + local
  `replace ../gen`, `replace ../types`).
- `eval` — requires `types` (currently `v0.0.0` + local `replace ../types`).

A bare `v0.1.0` tag versions only a root module and does **not** version
subdirectory modules; each module must be tagged with its path prefix
([go.dev/doc/modules/managing-source](https://go.dev/doc/modules/managing-source),
[golang/go#36014](https://github.com/golang/go/issues/36014)). `go.work` is
local-only and does **not** solve release ordering — consumers resolve via each
module's `go.mod`, so dependency modules must be tagged first and dependents'
`require` lines bumped before the dependents are tagged
([Go workspace proposal](https://go.googlesource.com/proposal/+/master/design/45713-workspace.md)).
Untagged inter-module requires fall back to pseudo-versions that look unreleased.

**Release order:**

1. Tag the leaves on the release commit: `types/v0.1.0` and `gen/v0.1.0`
   (independent; either order).
2. Bump `require` lines to the new tags: `harness/go.mod`
   (`gen v0.1.0`, `types v0.1.0`), `eval/go.mod` (`types v0.1.0`). Decide the
   `replace ../` directives: they are ignored by downstream consumers (replace
   only applies to a main module's own build) and may stay for local dev, but
   removing them is the cleaner published posture and forces a true
   tag-resolution build.
3. Per dependent module, verify published resolution **before** tagging:
   `GOFLAGS=-mod=mod GOWORK=off go build ./...` (with the `replace` removed, or
   pointed at the tag) — `GOWORK=off` so the workspace cannot mask a missing tag.
4. Tag `harness/v0.1.0` and `eval/v0.1.0` on the commit carrying the bumped
   `require` lines.

Commit-then-tag caveat for evidence (Section 5): the evidence commit's
`testedCommit` is its parent; the tag points at the evidence commit.

### 6.2 Release workflow touchpoints

`release.yml` triggers on `v*.*.*`, builds a 4-cell GOOS/GOARCH matrix
(`linux/amd64`, `linux/arm64`, `darwin/amd64`, `darwin/arm64`), marks hyphenated
tags as prereleases (`release.yml:687`), dual-publishes to ghcr.io + GAR, and
generates SBOMs. **It has zero executions ever.** Push `v0.1.0-rc1` first — a
prerelease tag exercises the entire pipeline (matrix, SBOM upload, dual-publish,
prerelease marking) and is the natural release-candidate step that closes
deployment-06/-07/-08/-16 before the final tags. Add the changelog evidence hook
(Section 5.4) in the same change. Note GoReleaser's single-tag model fits
binaries, not the multi-module library tagging above — the per-module tags are a
manual/`just`-scripted step, distinct from the binary release.

### 6.3 The 0.x alpha contract (release notes / `VERSIONING.md`)

Per Go's module guidance, v0 modules are "experimental"; stability promises
begin at v1 ([go.dev/doc/modules/developing](https://go.dev/doc/modules/developing),
[go.dev/blog/publishing-go-modules](https://go.dev/blog/publishing-go-modules)).
Quote SemVer item 4 verbatim:

> "Major version zero (0.y.z) is for initial development. Anything MAY change at
> any time. The public API SHOULD NOT be considered stable."
> ([semver.org item 4](https://semver.org/#spec-item-4))

Adopt an OpenTelemetry-style per-surface stability table in a root
`VERSIONING.md` ([otel versioning-and-stability](https://opentelemetry.io/docs/specs/otel/versioning-and-stability/)).
Recommended v0.1 declaration:

| Surface | v0.1 stability | Notes |
|---|---|---|
| CLI flags | alpha | may change between 0.x |
| `RunConfig` schema | alpha | pre-1.0; clean breaks preferred over shims |
| gRPC `harness.proto` contract | alpha | `stirrup job` control-plane API |
| `harness/harnessapi` Go API | alpha | `internal/` is private and not API |
| Security invariants (rings, Rule of Two, secret handling) | enforced | structurally tested; not a stability promise but a safety contract |

**v0.1 promises (CI-proven):** the unit/validation baseline of Section 1 (107
verified claims); planning-default safe-by-default invocation; secret

**v0.1 explicitly disclaims (must appear in the notes):** live conformance for

## 8. Appendix: methodology

This audit was produced in four stages:

1. **Claim extraction.** Falsifiable functionality claims were lifted from the
   shipped docs (`docs/architecture.md`, `configuration.md`, `providers.md`,
   `provider-quirks.md`, `security.md`, `safety-rings.md`, `executors/k8s.md`,
   `deployment.md`, `cloud-run-jobs.md`, `container-publishing.md`,
   `observability-cloud.md`, `trace-inspection.md`, `batch.md`,
   `credential-federation.md`, `anthropic-wif.md`, `azure-workload-identity.md`,
   `eval.md`, `philosophy.md`). Draft specs, third-party behaviour, and pure
   architectural narrative were excluded. Result: 161 claims across 9 areas.

2. **Evidence mapping.** Each claim was mapped to evidence at `file:line` (unit
   tests, code), `workflow + run date` (CI/smoke), or local script output, and
   classified `verified` / `partial` / `unverified` with an explicit gap.

3. **Adversarial audit.** A skeptic re-checked each classification. Two were
   overturned: `core-17` (verified -> partial; scrubber overclaim) and
   `executors-10` (stays partial, but the claim text is factually wrong). This
   plan trusts the skeptic's correction in both cases.

4. **Baseline + synthesis.** The local test baseline (Section 1, Phase A0) was
   executed independently; the collation, tagging, and alpha-contract guidance
   draw on the collation-design research and the industry-practice brief cited
   inline.
